ODB+ Documentation

About this documentation

This guide covers ODB+ setup, OBD-II protocol fundamentals, command reference, and troubleshooting. Written for both beginners and experienced automotive diagnosticians.

ODB+ is a professional-grade Android OBD-II diagnostic application. It connects to your vehicle's ECU via Bluetooth or WiFi OBD-II adapters and provides real-time sensor data, error code reading and clearing, AI-powered diagnostic analysis, and raw terminal access.

Requirements

  • Android 8.0 (Oreo) or higher
  • ELM327-compatible OBD-II adapter (Bluetooth Classic or WiFi)
  • Vehicle with OBD-II port (all cars sold in the US from 1996+, EU from 2001+)
  • Bluetooth permission granted in Android settings

Recommended adapters

Genuine ELM327 v2.1+ adapters work best. ODB+ also supports STN1100/STN2100 and ESP32-based adapters through its UOAPL (Universal OBD-II Adapter Protocol Layer). Avoid cheap clones that advertise v2.1 but actually run v1.5 firmware.

Installation

  1. Download ODB+ from the Google Play Store or install the APK directly
  2. Grant Bluetooth and Location permissions when prompted
  3. Plug your OBD-II adapter into the vehicle's diagnostic port (usually under the dash)
  4. Turn the vehicle's ignition to ON (engine can be off for code reading, must be running for live data)

First Connection

On first launch, ODB+ will scan for available Bluetooth OBD-II adapters. Select your adapter from the list. The app will then:

  1. Establish Bluetooth connection to the adapter
  2. Send initialization commands (ATZ, ATE0, ATL0, ATS1)
  3. Fingerprint the adapter to identify its capabilities
  4. Detect the vehicle's OBD-II protocol
  5. Discover supported PIDs
Initialization Sequence 
// 1. Reset adapter
> ATZ
ELM327 v2.3

// 2. Configure adapter
> ATE0          // Echo off
OK
> ATL0          // Linefeeds off
OK
> ATS1          // Spaces on
OK
> ATH1          // Headers on
OK

// 3. Detect protocol
> ATDP
AUTO, ISO 14230-4 (KWP FAST)

// 4. Query supported PIDs
> 0100
84 F1 10 41 00 BE 3E B8 11
// Bitmap: PIDs 01-20 supported mask

Important

Do not send ATSP0 before the first PID query. This resets the adapter's stored protocol hint and forces a full protocol scan, which can fail on some vehicles. ODB+ sends 0100 directly to leverage the adapter's last-known protocol.

OBD-II Protocols

OBD-II defines five communication protocols. Your vehicle uses exactly one. ODB+ auto-detects the correct protocol during connection.

Protocol AT Command Speed Common Vehicles
ISO 15765-4 (CAN 11-bit) ATSP6 500 kbps Most 2008+ vehicles worldwide
ISO 15765-4 (CAN 29-bit) ATSP7 500 kbps Some European vehicles
ISO 14230-4 (KWP2000 Fast) ATSP5 10.4 kbps 2003-2008 Asian and European
ISO 9141-2 ATSP3 10.4 kbps 1996-2004 European and Asian
SAE J1850 PWM ATSP1 41.6 kbps Ford vehicles (1996-2007)
SAE J1850 VPW ATSP2 10.4 kbps GM vehicles (1996-2007)

PID Reference

Parameter IDs (PIDs) are the standard codes used to request specific data from the vehicle's ECU. Service 01 PIDs request current (live) data. Below are the most commonly used PIDs.

PID Description Formula Unit Range
0x04 Calculated Engine Load A * 100 / 255 % 0 - 100
0x05 Engine Coolant Temperature A - 40 C -40 - 215
0x06 Short Term Fuel Trim (Bank 1) (A - 128) * 100 / 128 % -100 - 99.2
0x0B Intake Manifold Absolute Pressure A kPa 0 - 255
0x0C Engine RPM (A * 256 + B) / 4 RPM 0 - 16,383
0x0D Vehicle Speed A km/h 0 - 255
0x0F Intake Air Temperature A - 40 C -40 - 215
0x10 MAF Air Flow Rate (A * 256 + B) / 100 g/s 0 - 655.35
0x11 Throttle Position A * 100 / 255 % 0 - 100
0x14 O2 Sensor Voltage (Bank 1, Sensor 1) A / 200 V 0 - 1.275

AT Command Reference

AT commands configure the ELM327 adapter. They always start with "AT" and are not sent to the vehicle's ECU. These are the most useful commands.

Command Description Response
ATZ Reset adapter to defaults ELM327 vX.X
ATE0 / ATE1 Echo off / on OK
ATL0 / ATL1 Linefeeds off / on OK
ATS0 / ATS1 Spaces off / on in responses OK
ATH0 / ATH1 Headers off / on OK
ATDP Display current protocol name Protocol name string
ATSPn Set protocol (0=auto, 1-9=specific) OK
ATRV Read battery voltage XX.XV
ATI Print firmware version ELM327 vX.X
ATCAF0 / ATCAF1 CAN auto formatting off / on OK

Caution

Do not send ATCAF0 unless you understand CAN framing. With CAN auto formatting disabled, all CAN PID queries will return "NO DATA". ODB+ always re-enables ATCAF1 after adapter fingerprinting to prevent this issue.

DTC Format

Diagnostic Trouble Codes (DTCs) follow a standard 5-character format. The first character indicates the system, followed by 4 hex digits.

DTC Structure 
// DTC format: X0000
//             |||||
//             ||||+-- Fault index (00-99)
//             |||+--- Sub-system
//             ||+---- System (0-3)
//             |+----- Type: 0=SAE, 1=Manufacturer
//             +------ Category

P = Powertrain   (engine, transmission)
B = Body          (airbags, lighting, HVAC)
C = Chassis       (ABS, steering, suspension)
U = Network       (CAN bus communication)

// Example: P0300
P  = Powertrain
0  = SAE standard code
3  = Ignition system / misfire
00 = Random/multiple cylinder misfire

ODB-II defines three DTC query services:

  • 03 -- Read stored (confirmed) DTCs. These have illuminated the MIL (Check Engine Light).
  • 07 -- Read pending DTCs. The fault has occurred but not enough times to confirm.
  • 0A -- Read permanent DTCs (CAN only). Cannot be cleared until the fault condition is resolved.

Terminal Usage

The Terminal screen provides direct access to the OBD-II bus. Type commands and see raw hex responses. Useful for advanced diagnostics, protocol debugging, and accessing manufacturer-specific data.

Common Workflows

Read and Clear DTCs 
// Read stored DTCs
> 03
84 F1 10 43 01 03 00 01 71 00
// Decoded: P0300, P0171

// Clear DTCs and reset MIL
> 04
84 F1 10 44
// DTCs cleared. MIL will turn off.

// Verify cleared
> 03
84 F1 10 43 00
// No DTCs stored
Request Vehicle Information 
// Request VIN (Service 09, PID 02)
> 0902
84 F1 10 49 02 01 57 46 30
84 F1 10 49 02 02 58 58 47
84 F1 10 49 02 03 43 44 35
// Multi-frame ISO-TP response
// Decoded VIN: WF0XXGCD5...

// Request calibration ID
> 0904
// ECU calibration identifier

Adapter Compatibility

ODB+ uses its UOAPL (Universal OBD-II Adapter Protocol Layer) to support multiple adapter families transparently.

Adapter Family Driver Features Status
ELM327 v2.x ElmDriver Full protocol support, AT commands Supported
ELM327 v1.5 ElmDriver Basic PIDs, limited protocol detection Limited
STN1100 / STN2100 StnDriver Enhanced PIDs, faster CAN, ST commands Supported
ESP32-based Esp32Driver Custom firmware, direct CAN access Supported

Troubleshooting

Connection Issues

  • "UNABLE TO CONNECT" -- The adapter could not establish protocol communication. Try turning the engine on (running), then reconnect. Some vehicles require the engine running for bus initialization.
  • "BUS INIT: ERROR" -- Protocol bus initialization failed. This can happen if the wrong protocol was selected. ODB+ will automatically try other protocols. If persistent, try a full adapter reset.
  • "NO DATA" on all PIDs -- Check that ATCAF1 is enabled (CAN auto formatting). Also verify the adapter is genuinely connected to the vehicle and not just powered by the OBD port.
  • Bluetooth disconnects frequently -- Move the phone closer to the adapter. Some cheap adapters have weak Bluetooth radios. Ensure no other apps are competing for the Bluetooth connection.

Data Issues

  • RPM reads 0 with engine running -- Some KWP2000 vehicles require ATS1 (spaces on) for correct response parsing. ODB+ enables this by default, but check if a terminal command disabled it.
  • Temperature reads -40 C -- The sensor is not responding. This is the default "no data" value (0x00 - 40 = -40). The ECU may not support this PID, or the sensor is disconnected.
  • Supported PID count shows 0 -- The initial 0100 query failed. This can happen if ATCAF0 was sent. Reconnect to reset the adapter configuration.
