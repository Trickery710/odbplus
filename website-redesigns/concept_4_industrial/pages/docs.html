ODB+ Diagnostic System -- Technical Manual Rev. 2.4
This guide covers setup and operation of the ODB+ diagnostic system. It is intended for automotive technicians, fleet maintenance personnel, and vehicle enthusiasts operating ELM327-compatible OBD-II adapters. Read all sections before first use. Refer to the Troubleshooting section for common issues.
Section 01

Getting Started

Before connecting to a vehicle, verify that your equipment meets the following minimum requirements. Operating below these specifications may result in degraded performance or connection failures.

System Requirements

  • Android 8.0 (Oreo) or higher -- required for Bluetooth SPP and BLE support
  • 2 GB RAM minimum -- 4 GB recommended for AI diagnostics and session recording
  • 45 MB storage -- additional space required for session logs and exports
  • Bluetooth Classic (SPP) or Bluetooth Low Energy (BLE) -- adapter dependent
  • WiFi (optional) -- for TCP socket adapters on port 35000
  • ELM327 v1.3+ compatible adapter -- supports all major clones and STN chipsets
  • OBD-II compliant vehicle -- all US vehicles 1996+ and EU vehicles 2001+

Supported Adapters

  • ELM327 -- all versions v1.3 and above, including Chinese clones
  • STN1110 / STN2120 -- full extended command support
  • OBDLink -- MX+, LX, EX, SX series
  • ESP32-based -- WiFi adapters with TCP socket interface
  • Vgate / Veepeak / BAFX -- Bluetooth ELM327-compatible units
Section 02

Adapter Setup

The OBD-II diagnostic link connector (DLC) is a standardized 16-pin port located under the dashboard on the driver's side, typically within arm's reach of the steering column. Some vehicles mount it behind a removable panel or above the pedals.

Bluetooth Pairing Procedure

  1. [01]
    Insert the Adapter

    Plug your ELM327 adapter into the vehicle's 16-pin OBD-II DLC port. The adapter should power on automatically when the vehicle ignition is in the ON or ACC position. Confirm the adapter's LED indicator is lit.

  2. [02]
    Open Bluetooth Settings

    On your Android device, navigate to Settings > Bluetooth. Enable Bluetooth if it is not already active. Tap "Pair new device" or "Scan for devices."

  3. [03]
    Select Your Adapter

    Your adapter will appear in the scan results with a name such as "OBDII", "OBDLink", "Vlink", "Android-Vlink", or similar. Tap the device name to initiate pairing.

  4. [04]
    Enter the Pairing PIN

    When prompted, enter the pairing PIN. The default PIN for most ELM327 adapters is 1234 or 0000. Check your adapter's documentation if these do not work.

  5. [05]
    Launch ODB+ and Connect

    Open the ODB+ app. Navigate to the connection screen and select your paired adapter from the device list. Tap "Connect." The app will fingerprint your adapter, detect its capabilities, and negotiate the vehicle's OBD-II protocol automatically.

Section 03

First Connection

On first connection, ODB+ performs an automated initialization sequence. The adapter is reset, echo and linefeeds are disabled, and the vehicle's OBD-II protocol is auto-detected. This process takes 2-5 seconds for CAN vehicles and up to 15 seconds for older KWP2000/ISO 9141 protocols.

What happens during initialization: The app sends a series of AT commands to configure the adapter, then issues a supported-PID query (0100) to discover which sensor parameters your vehicle's ECU reports. This PID bitmap determines which dashboard gauges and live data parameters are available.

// Step 1: Reset adapter to factory defaults ODB+> ATZ ELM327 v2.3 // Step 2: Disable echo, linefeeds; enable spaces and headers ODB+> ATE0 OK ODB+> ATL0 OK ODB+> ATS1 OK ODB+> ATH1 OK // Step 3: Query supported PIDs (triggers protocol negotiation) ODB+> 0100 84 F1 10 41 00 BE 3E B8 13 // Decoded: PIDs 01-20 bitmap = BE 3E B8 13 // Step 4: Confirm negotiated protocol ODB+> ATDP AUTO, ISO 14230-4 (KWP FAST)
Section 04

Live Data

The Live Data screen streams real-time sensor readings from your vehicle's ECU. Data is refreshed at the maximum rate supported by your vehicle's protocol -- approximately 200ms for CAN, 400ms for KWP2000, and 600ms for ISO 9141.

View modes: Analog gauges for at-a-glance monitoring, digital readouts for precision values, time-series graphs for trend analysis, and raw hex mode for protocol-level inspection. You can configure custom dashboards with 1-8 parameters.

Performance tip: Monitoring fewer PIDs simultaneously increases refresh rate per parameter. For diagnostic accuracy, limit active parameters to 4-6 when using non-CAN protocols.

// Read engine RPM ODB+> 010C 84 F1 10 41 0C 0B 28 // Formula: (0x0B * 256 + 0x28) / 4 = 2,842 RPM // Read coolant temperature ODB+> 0105 84 F1 10 41 05 67 // Formula: 0x67 - 40 = 63C (145F) // Read vehicle speed ODB+> 010D 84 F1 10 41 0D 3E // Formula: 0x3E = 62 km/h (38 mph) // Read throttle position ODB+> 0111 84 F1 10 41 11 3A // Formula: (0x3A * 100) / 255 = 22.7%
Section 05

Trouble Codes

ODB+ reads and clears Diagnostic Trouble Codes (DTCs) stored in the vehicle's ECU. The database includes over 18,000 codes: generic OBD-II codes (P0xxx, B0xxx, C0xxx, U0xxx) and manufacturer-specific codes for 50+ automakers. Each code entry includes a description, probable causes, repair procedures, and severity rating.

Code types: Stored codes (confirmed faults), pending codes (intermittent faults that have not yet set a check engine light), and permanent codes (codes that cannot be cleared until the underlying condition is repaired and drive cycles are completed).

// Read stored DTCs (Mode 03) ODB+> 03 84 F1 10 43 03 00 01 71 00 00 // Decoded: P0300 - Random/Multiple Cylinder Misfire // P0171 - System Too Lean (Bank 1) // Read pending DTCs (Mode 07) ODB+> 07 84 F1 10 47 01 28 00 00 00 00 // Decoded: P0128 - Coolant Thermostat Below Threshold // Clear all DTCs and freeze frame data (Mode 04) ODB+> 04 44 // DTCs cleared. Check engine light extinguished.
Section 06

Terminal Mode

Terminal Mode provides direct command-line access to the ELM327 adapter. You can send any AT configuration command, raw OBD-II PID request, or hex-level protocol data. Responses are displayed unfiltered with header bytes, timestamps, and protocol metadata.

Warning: Sending incorrect AT commands can misconfigure the adapter and break communication with the vehicle ECU. Use ATZ to reset to defaults if you encounter issues.

// Check adapter firmware version ODB+> ATI ELM327 v2.3 // Read battery voltage ODB+> ATRV 12.6V // Display current protocol ODB+> ATDP AUTO, ISO 14230-4 (KWP FAST) // Manually select CAN 11-bit 500kbps ODB+> ATSP6 OK // Enable CAN auto-formatting ODB+> ATCAF1 OK
Section 07

AI Diagnostics

The AI diagnostic assistant has real-time access to your vehicle's ECU data. Describe your symptoms in natural language -- "rough idle at cold start", "hesitation under load", "P0171 after highway driving" -- and the AI correlates your description with live sensor readings to identify probable causes.

Capabilities: Fault code interpretation with vehicle-specific context, sensor cross-correlation (e.g., linking lean codes with MAF and fuel trim data), severity assessment (drive/do not drive recommendations), repair procedure suggestions ranked by probability, and parts cost estimates.

Requirement: AI diagnostics requires an active internet connection. All other features function offline.

Section 08

Session Logger

The Session Logger records complete diagnostic sessions with timestamped sensor data, DTCs, and command logs. Sessions can be replayed to review data trends, compared before/after repair to verify fixes, and exported as CSV or JSON for external analysis.

Recording: Tap the record button on any Live Data screen. All active PID readings are captured at the current refresh rate. Session files are stored locally and can be shared via the community forums or exported to cloud storage.

Replay: Load any saved session from the Session History screen. Data plays back at original timing with full gauge and graph visualization.

Section 09

Protocol Specifications

ODB+ automatically detects and negotiates the correct OBD-II protocol for your vehicle. Manual protocol selection is available in Terminal Mode via ATSP commands. The following table lists all supported protocols, their standards, initialization methods, and typical vehicle applications.

Protocol Standard Init Method Typical Vehicles
SAE J1850 PWM J1850 41.6 kbps pulse-width Ford vehicles (pre-2008)
SAE J1850 VPW J1850 10.4 kbps variable pulse GM vehicles (pre-2008)
ISO 9141-2 ISO 9141 5-baud slow init (~2.7s) Chrysler, Asian imports (pre-2004)
ISO 14230-4 KWP (Slow) KWP2000 5-baud slow init European vehicles (pre-2005)
ISO 14230-4 KWP (Fast) KWP2000 Fast init (~200ms) European, Korean vehicles
ISO 15765-4 CAN (11-bit, 500k) CAN CAN bus arbitration Most 2008+ US vehicles
ISO 15765-4 CAN (29-bit, 500k) CAN CAN bus arbitration Heavy duty, some European
ISO 15765-4 CAN (11-bit, 250k) CAN CAN bus arbitration Some older CAN vehicles
ISO 15765-4 CAN (29-bit, 250k) CAN CAN bus arbitration Some older CAN vehicles

Auto-detection order: ODB+ queries the adapter's stored protocol hint first (via the last-used protocol memory). If no hint is available, it falls back to a priority scan: CAN 11-bit 500k first (most common), then KWP Fast, then remaining protocols in descending frequency order. This minimizes connection time for the vast majority of vehicles.

Section 10

Troubleshooting

Frequently Asked Questions

Why does my adapter show "UNABLE TO CONNECT"?
This occurs when the ELM327 cannot negotiate a protocol with the vehicle's ECU. Common causes: the ignition is not in the ON position, the adapter is not fully seated in the DLC port, or the adapter's protocol hint is stale. Try: turn the ignition fully ON (engine running is best), reseat the adapter, then open Terminal Mode and send ATZ followed by 0100. If the adapter retains the correct protocol hint, it will connect. If not, send ATSP0 to force a full protocol scan.
How do I know which OBD-II protocol my vehicle uses?
After a successful connection, open Terminal Mode and send ATDP. The adapter will respond with the active protocol name (e.g., "ISO 14230-4 (KWP FAST)" or "ISO 15765-4 (CAN 11/500)"). As a general rule: most US vehicles 2008+ use CAN, European vehicles 2001-2007 commonly use KWP2000, and older vehicles may use ISO 9141 or J1850.
Can I use ODB+ with a WiFi adapter?
Yes. ODB+ supports WiFi ELM327 adapters that provide a TCP socket interface (typically on port 35000). In the connection settings, select "WiFi" as the connection type and enter the adapter's IP address (usually 192.168.0.10). Note that WiFi adapters require your phone to disconnect from your regular WiFi network and connect to the adapter's network.
Why are some PIDs showing "No Data"?
Not all vehicles support all OBD-II PIDs. The ECU reports which PIDs it supports via the 0100/0120/0140 bitmap queries. If a PID returns "No Data," your vehicle's ECU does not implement that parameter. This is normal -- vehicles typically support 15-30 of the 80+ standardized PIDs. You can verify supported PIDs by sending 0100 in Terminal Mode and decoding the bitmap response.
Is ODB+ safe to use while driving?
ODB+ reads sensor data passively. It does not write to the ECU or modify any vehicle parameters (except when you explicitly clear DTCs via Mode 04). Reading live data while driving is safe from a vehicle standpoint. However, do not interact with the app while driving. Use the HUD overlay for hands-free monitoring or have a passenger operate the app.
